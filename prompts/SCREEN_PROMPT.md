# BidOps - SCREEN_PROMPT

## 목적
이 프롬프트는 화면 기획, 와이어프레임, UI 설계, 퍼블리싱 작업을 수행할 때 사용하는 기준 프롬프트다.

목표는 예쁜 화면이 아니라, **업무형 B2B 시스템으로서 명확하고 신뢰감 있는 UI**를 만드는 것이다.

---

## 제품 특성
- 공공/민간 입찰 및 제안 업무용 시스템
- 문서 분석, 요구사항 검토, 누락 탐지, 제안 운영 지원이 핵심
- 사용자는 PM, 제안 실무자, 관리자
- 감성형 서비스가 아니라 정보 밀도가 높은 업무형 SaaS

---

## UI 원칙
1. 예쁨보다 정보 구조를 우선한다.
2. 한 화면에서 현재 상태와 다음 액션이 분명해야 한다.
3. 표, 필터, 상태 배지, 근거 링크가 핵심이다.
4. 사용자는 AI의 답보다 근거와 검토 상태를 먼저 확인해야 한다.
5. 검토/승인 흐름이 명확해야 한다.

---

## 디자인 톤
- 깔끔하고 신뢰감 있는 B2B 톤
- 과한 색상, 과한 애니메이션 금지
- 카드 + 테이블 + 패널 중심
- 좌측 네비게이션 + 상단 헤더 구조 우선
- "노션 + 어드민 대시보드 + 문서검토툴" 느낌

---

## 핵심 화면 대상
아래 화면을 우선 설계 대상으로 본다.
- 프로젝트 목록
- 프로젝트 상세
- 문서 업로드
- RFP 분석 결과 대시보드
- 요구사항 목록
- 요구사항 상세 분석
- 누락/증빙 체크리스트
- 검토/승인 화면

---

## 컴포넌트 설계 원칙
- 버튼 종류는 Primary / Secondary / Ghost / Danger 정도로 제한
- 상태 배지는 명확한 enum 기반으로 설계
- 테이블은 정렬, 필터, 검색, 상태표시가 기본
- 상세 패널은 원문 근거와 AI 분석 결과를 나란히 비교 가능해야 함
- 페이지 내 CTA는 1~2개만 강하게 강조

---

## 퍼블리싱 방향
- Next.js + Tailwind CSS 기반
- shadcn/ui 스타일의 재사용형 컴포넌트 우선
- 화면별 신규 CSS 남발 금지
- 레이아웃, 카드, 테이블, 폼, 모달을 공통화

---

## UX 금지사항
- 한 화면에 너무 많은 강조색 사용 금지
- 원문 근거 없는 AI 결과 단독 노출 금지
- 상태값이 텍스트만 있고 시각적으로 안 보이는 UI 금지
- 버튼 우선순위가 뒤섞인 화면 금지

---

## 화면 설계 시 추가 지시
- "이 화면이 없으면 사용자가 어디서 막히는가"를 먼저 생각한다.
- 분석 결과 화면에서는 반드시 원문 근거, 신뢰 상태, 검토 액션이 함께 보이게 한다.
- 단순 카드 나열이 아니라 실제 업무 흐름이 이어지도록 설계한다.
