# BidOps Claude Code Instructions

이 프로젝트는 BidOps다. 공공/민간 입찰·제안 업무를 위한 근거 기반 RFP 분석 및 수주 운영 시스템을 개발한다.

## 최우선 원칙
1. 이 프로젝트를 제안서 자동작성기가 아니라 근거 기반 요구사항 분석 시스템으로 구현한다.
2. 정확성, 추적 가능성, 검토 가능성을 속도보다 우선한다.
3. 문서 분석 결과는 항상 원문 근거와 연결 가능해야 한다.
4. AI 해석과 사실 추출을 구분할 수 있는 데이터 구조를 유지한다.
5. 한 번에 큰 수정 금지. 항상 작은 단위로 구현하고 검증한다.

## 구현 원칙
- 프론트엔드: Next.js
- 백엔드: Spring Boot
- AI 워커: Python
- OCR/문서 구조화: Azure Document Intelligence
- RFP 분석 주엔진: OpenAI GPT-5.4
- 신뢰성 보강: 규칙 엔진 + 사람 검토

## 개발 작업 원칙
- 기능 구현 전 관련 문서를 먼저 읽는다.
- API, DB, UI는 docs/ 와 prompts/ 문서를 기준으로 일관되게 작성한다.
- 추측으로 구현하지 말고, 불명확하면 TODO 또는 질문 포인트를 남긴다.
- 테스트 가능한 단위로 나누어 구현한다.
- 임의의 범위 확장 금지.

## 1. 문서 목적
이 문서는 Claude Code가 BidOps 프로젝트를 일관되게 이해하고 개발하기 위한 작업 기준 문서다.
이 문서를 읽는 모든 AI는 프로젝트 목적, 기술 방향, 구조, 작업 규칙, 금지사항, 산출 기준을 동일하게 적용해야 한다.

이 문서는 다음 목적을 가진다.
- 프로젝트 방향 흔들림 방지
- 코드 품질 일관성 유지
- 기능 개발 순서 고정
- 과도한 자동화/과대 구현 방지
- RFP 분석 신뢰성 우선 원칙 유지

## 2. 프로젝트 요약
### 2.1 제품명
BidOps

### 2.2 한 줄 정의
BidOps는 공공/민간 입찰·제안 업무에서 RFP를 구조적으로 분석하고, 요구사항 누락을 방지하며, 제안서 작성과 수주 운영을 지원하는 근거 기반 문서 운영 AI OS다.

### 2.3 제품 철학
이 제품의 본질은 문장 생성이 아니라 **검증 가능한 분석**이다.

다음 원칙을 항상 우선한다.
1. 원문 근거가 없는 확정적 판단 금지
2. 자유서술보다 구조화 출력 우선
3. 누락 탐지, 충돌 탐지, 모호성 탐지가 핵심 가치
4. AI 단독 결론보다 사람 검토 가능한 결과가 중요
5. 화려한 UI보다 신뢰 가능한 분석 흐름이 중요

## 3. 현재 고정된 기술 방향
### 3.1 개발 조합
- 개발: Claude Code + IntelliJ
- OCR / 문서 구조화: Azure Document Intelligence
- RFP 분석 주엔진: OpenAI GPT-5.4
- 검증: 규칙 엔진 + 사람 검토

### 3.2 애플리케이션 구조
- 프론트엔드: Next.js
- 백엔드: Spring Boot
- AI 워커: Python
- DB: PostgreSQL
- 파일 저장: S3 또는 동급 객체 스토리지
- 배포: AWS App Runner + RDS + S3 또는 동급 구조

### 3.3 아키텍처 철학
- 모놀리식 API + 분리된 AI 워커 구조를 우선한다.
- 초기에는 MSA, Kubernetes, 복잡한 이벤트 버스를 도입하지 않는다.
- 작은 단위 기능을 먼저 완성하고, 실제 동작하는 산출물을 우선한다.
- 멀티 에이전트보다 결정형 워크플로우가 우선이다.

### 4.1 우선순위

## 5. Claude Code의 역할
Claude Code는 이 프로젝트에서 다음 역할을 수행한다.
- 지정된 범위의 코드 구현
- 리팩토링
- 테스트 코드 작성
- 패키지/모듈 정리
- API/DTO/엔티티 생성 보조
- 프론트 컴포넌트 구현
- 반복 작업 자동화
- 문서화

Claude Code는 다음 역할을 수행하지 않는다.
- 제품 방향 임의 변경
- 범위 확대
- 미확정 기술 도입
- 신뢰성 원칙 완화
- 요구사항 분석 기준 임의 변경

---

## 6. 작업 방식 규칙
### 6.1 기본 원칙
- 한 번에 한 모듈만 작업한다.
- 항상 작은 단위로 구현한다.
- 먼저 구조를 만들고, 그 다음 기능을 붙인다.
- 먼저 작동하는 최소 기능을 만들고, 이후 정제한다.
- 가정이 필요한 경우 명시적으로 TODO 또는 FIXME를 남긴다.
- 임의 추정으로 복잡한 비즈니스 규칙을 만들지 않는다.

### 6.2 금지사항
- 한 번에 대규모 리팩토링 금지
- 전체 구조를 뒤엎는 변경 금지
- 사용자가 합의하지 않은 라이브러리 추가 금지
- 불필요한 추상화 금지
- 조기 최적화 금지
- 테스트 없이 핵심 로직 변경 금지

### 6.3 수정 원칙
기존 파일 수정 시에는 다음을 우선한다.
1. 최소 변경
2. 영향 범위 명확화
3. 기존 스타일 유지
4. 테스트 보강
5. 변경 이유 문서화

## 7. 프로젝트 개발 단계
### Phase 1. 기반 구축
- 인증 없는 로컬 프로토타입도 허용
- 프로젝트 생성
- 문서 업로드
- 파일 메타데이터 저장
- 기본 화면 뼈대

### Phase 2. 문서 분석 기반
- PDF 업로드 후 파싱
- OCR / 레이아웃 구조 저장
- 페이지 단위 원문 저장
- 문서/페이지/섹션 메타데이터 구조화

### Phase 3. 핵심 AI
- 요구사항 자동 분해
- 구조화 JSON 저장
- 요구사항 목록/상세 UI
- 근거 페이지 연결

### Phase 4. 누락 탐지
- 필수 제출물 체크리스트
- 증빙 누락 탐지
- 질의 필요 여부 표시
- 충돌/모호성 표시

### Phase 5. 운영 흐름
- 검토/승인 상태
- 수정 이력
- 유사 제안 검색
- 산출물 관리

## 13. UI/UX 구현 원칙
### 13.1 화면 철학
- 예쁜 화면보다 업무 효율 우선
- 문서와 요구사항 비교가 쉬워야 함
- 필터와 상태가 직관적이어야 함
- 위험 항목이 눈에 잘 보여야 함

### 13.2 디자인 원칙
- 깔끔하고 신뢰감 있는 톤
- 대시보드형 B2B 업무 도구 스타일
- 카드 + 표 + 상세패널 조합
- 과한 애니메이션 금지
- 색은 상태/위험도 강조에만 사용

### 13.3 프론트 구현 원칙
- Tailwind + 컴포넌트 기반 구현
- 공통 UI 패턴 재사용
- 비즈니스 로직은 프론트에 과도하게 넣지 않음
- 서버 상태와 로컬 상태 분리

## 17. Claude Code 작업 요청 형식
Claude Code에 작업 요청할 때는 다음 구조를 권장한다.

1. 작업 목적
2. 수정 대상 파일
3. 구현 범위
4. 세부 요구사항
5. 금지사항
6. 완료 기준
7. 테스트 기준

예시:
- 목적: Project 생성 API 추가
- 대상: project 패키지
- 범위: controller/service/domain/repository/dto
- 요구사항: name, clientName, bidType 저장
- 금지사항: 인증, 파일 업로드, 복잡한 예외처리 추가 금지
- 완료 기준: 생성/조회 API 동작
- 테스트 기준: service test + controller test

## 18. 코드 스타일 가이드
- 이름은 의미가 분명해야 한다.
- 축약어 남용 금지
- DTO / Entity / Response 역할 분리
- 매직넘버 금지
- enum 적극 활용
- 주석은 왜 필요한지 설명할 때만 사용
- 코드로 표현 가능한 것은 주석보다 코드로 표현
- null 처리 명확화
- 예외 메시지 일관성 유지

## 19. 오픈 이슈 관리 원칙
아직 미정인 사안은 억지로 구현하지 말고 다음 방식 중 하나를 사용한다.
- TODO

- DECISION NEEDED
- ASSUMPTION

## 20. 이 프로젝트에서 가장 중요한 최종 문장
BidOps의 성공 조건은 "멋지게 써주는 AI"가 아니라,
