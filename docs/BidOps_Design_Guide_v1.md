# BidOps Design Guide v1

> 목적: Claude Code가 BidOps 프론트 화면을 수정/구현할 때 참조할 수 있는 공통 디자인 기준서
> 
> 범위: 로그인, 대시보드, 프로젝트, 문서, 요구사항, 체크리스트, 관리자 화면 전반

---

## 1. 제품 디자인 목표

BidOps는 "예쁜 화면"보다 **신뢰 가능한 업무 처리 화면**을 만드는 것이 우선이다.

핵심 목표:
- 근거 기반 분석 시스템처럼 보여야 한다.
- AI 분석 결과와 사람 검토 결과가 명확히 분리되어야 한다.
- 현재 상태, 다음 액션, 원문 근거 위치가 즉시 보여야 한다.
- 공공/민간 입찰 실무자가 써도 어색하지 않은 B2B SaaS 톤을 유지해야 한다.

핵심 키워드:
- 신뢰감
- 정돈감
- 명료성
- 업무성
- 차분함

지양:
- 소비자 앱 같은 감성형 디자인
- 과한 애니메이션
- 과한 그림자
- 장식성 위주의 컬러 사용
- 카드 남발
- 정보보다 비주얼이 앞서는 구성

---

## 2. 전체 UX 원칙

1. **정확성 > 화려함**
2. 정보 밀도가 높아도 구조는 명확해야 한다.
3. 사용자는 항상 아래 4가지를 빠르게 이해할 수 있어야 한다.
   - 지금 보고 있는 정보가 무엇인지
   - 이 정보가 AI 분석인지, 사람 검토인지
   - 원문 근거가 어디 있는지
   - 다음 액션이 무엇인지
4. 중요한 정보는 문장보다 **표, 배지, 상태, 그룹 구조**로 드러낸다.
5. 카드 UI는 요약용으로만 사용하고, 핵심 업무는 테이블/상세 패널 중심으로 설계한다.
6. 데스크톱 우선으로 설계하되, 태블릿 이하에서는 패널 수를 줄여 대응한다.

## 3. 레이아웃 원칙

### 3.1 앱 공통 구조
- **좌측 네비게이션**: 1차 메뉴
- **상단 컨텍스트 바**: 현재 프로젝트/페이지 제목/주요 액션
- **본문 작업 영역**: KPI 카드 + 필터 바 + 테이블/상세 패널

### 3.2 1차 메뉴 권장 구조
- 대시보드
- 프로젝트
- 문서함
- 요구사항
- 체크리스트
- 질의응답
- 산출물
- 관리자

### 3.3 상세 화면 구조 원칙
리스트/테이블과 연결되는 상세 화면은 다음 구조를 우선한다.
- 좌측: 원문/PDF/근거
- 중앙: 구조화 분석 정보
- 우측: 검토 상태/이력/액션

### 3.4 반응형 원칙
- Desktop: 3패널 허용
- Tablet: 2패널 전환
- Mobile: 1패널 + 탭/아코디언 구조

- `gray-050: #F8FAFC`

#### Semantic
- `success: #0F9F6E`
- `warning: #D97706`
- `danger: #DC2626`
- `info: #2563EB`

#### Surface
- `app-bg: #F6F8FC`
- `surface: #FFFFFF`
- `surface-muted: #F9FBFF`
- `border: #E5EAF2`

### 4.2 상태 색상 원칙
- 기본/대기: 회색/네이비
- 진행중: 파랑
- 승인/완료: 초록
- 검토 필요: 주황
- 위험/누락: 빨강
- 보류: 회색
- 질의 필요: 인디고/퍼플 계열

### 4.3 Typography
- 페이지 타이틀: `28px / 700`
- 섹션 타이틀: `20px / 700`
- 카드 타이틀: `16px / 600`
- 본문: `14px / 400~500`
- 보조 설명: `13px / 400`
- 캡션/라벨: `12px / 500`

문장 스타일:
- 짧고 명확한 업무형 문장 사용
- 마케팅 문구 금지
- 버튼/라벨은 동사형 또는 설명형으로 단문 사용

### 4.4 Radius / Shadow / Border
- 기본 radius: `10px`
- 큰 카드 radius: `14px`
- 그림자: 매우 약한 1단계만 허용
- border는 적극 사용, shadow는 최소화

### 4.5 Spacing
- 스케일: `4 / 8 / 12 / 16 / 20 / 24 / 32 / 40`
- 페이지 기본 패딩: `24px`
- 카드 내부 패딩: `20~24px`
- 입력 필드 높이: `44~48px`

## 5. 핵심 컴포넌트 가이드

### 5.1 버튼
종류:
- Primary: 현재 화면의 가장 중요한 액션
- Secondary: 보조 액션
- Tertiary/Text: 낮은 우선순위
- Danger: 삭제/위험 작업

원칙:
- 한 화면에 Primary는 보통 1개만 둔다.
- 버튼 텍스트는 짧고 명확한 동사형을 사용한다.
- 예: `로그인`, `분석 시작`, `검토 승인`, `근거 보기`, `저장`

금지:
- 한 화면에서 Primary 버튼 여러 개 남발
- 긴 문장 버튼

### 5.2 입력 필드
원칙:
- 라벨은 항상 필드 위에 배치
- placeholder는 예시용으로만 사용
- 기본 높이 48px
- focus 상태에서 1px accent border + 약한 focus ring 사용
- 배경색은 순백 또는 매우 옅은 회색으로 통일

### 5.3 Status Badge
BidOps에서 배지는 매우 중요하다.

대표 상태:
- 분석 완료
- 검토 필요
- 승인됨
- 누락 위험
- 질의 필요
- 확정 정보
- 추론 정보

원칙:
- 색만으로 상태를 전달하지 말고 텍스트도 함께 표시
- 같은 상태는 모든 화면에서 같은 색/패턴 사용

### 5.4 테이블
BidOps의 핵심 UI는 테이블이다.

원칙:
- 첫 열은 식별자/제목
- 상태 컬럼은 좁게
- 설명 컬럼은 넓게
- 정렬, 필터, 검색 기본 제공
- 행 클릭 시 상세 패널/상세 페이지 이동
- 고정 헤더 권장
- 다중 선택은 꼭 필요한 화면에서만 사용

### 5.5 필터 바
기본 구성:
- 키워드 검색
- 분류 필터
- 분석 상태 필터
- 검토 상태 필터
- Fact Level 필터
- 문서 유형 필터
- 정렬
- 초기화 버튼

### 5.6 카드
카드는 **요약용**이다.

허용 예시:
- 총 요구사항 수
- 검토 필요 수
- 누락 위험 수
- 질의 필요 수
- 문서 분석 상태

금지:
- 긴 설명을 카드 여러 장에 나눠 담기
- 핵심 업무를 카드 위주로 구현하기

### 5.7 상세 패널
포함 요소:
- 원문
- 분석 요약
- 검토 상태
- 근거 페이지
- 수정/승인 액션

원칙:
- 정보는 섹션별로 분리
- AI 분석 / 사람 검토를 시각적으로 구분
- 근거 이동 액션을 항상 상단에 가깝게 배치

## 6. 화면별 디자인 기준

## 6.1 로그인 화면

### 목표
단순 템플릿 로그인 화면이 아니라, **문서 운영형 B2B SaaS의 진입 화면**처럼 보여야 한다.

### 추천 구조
#### A안: 좌우 2단형
- 좌측: 브랜드/제품 설명
- 우측: 로그인 카드

좌측 내용:
- BidOps 로고
- 제품 한 줄 설명
- 핵심 가치 3개
  - 근거 기반 분석
  - 누락 방지 체크리스트
  - 사람 검토/승인 흐름

우측 내용:
- 이메일/비밀번호 폼
- 로그인 버튼
- 비밀번호 찾기
- 회원가입
- 오류 메시지

#### B안: 중앙 강화형
- 상단 브랜드 영역
- 중간 로그인 카드
- 하단 보조 설명

### 스타일 규칙
- 배경은 `app-bg`
- 카드: border + subtle shadow + radius 10
- 로고는 작고 정교하게
- 제목은 짧고 단단하게
- 설명은 1줄보다 2줄 구조 추천
- `비밀번호 보기`는 텍스트보다 아이콘 버튼 권장
- `비밀번호 찾기` 링크 추가 권장

### 추천 카피
- 제목: `BidOps`
- 설명: `근거 기반 RFP 분석과 누락 방지를 위한 문서 운영 시스템`
- 보조 설명: `프로젝트, 문서, 요구사항, 체크리스트를 한 흐름으로 관리합니다.`

---

## 6.2 대시보드

### 목적
사용자가 로그인 직후 현재 상황을 빠르게 파악하게 한다.

### 구성
- 상단: 사용자 인사 + 프로젝트 컨텍스트
- KPI 카드 4개
- 중단 2열
  - 좌: 요구사항 분포 / 누락 위험
  - 우: 최근 검토 / 질의 필요
- 하단: 주요 화면으로 이동하는 CTA

### 표시 우선순위
1. 프로젝트 수
2. 검토 필요 수
3. 누락 위험 수
4. 최근 활동

---

## 6.3 프로젝트 목록

### 목적
프로젝트를 빠르게 탐색하고 상태를 파악하게 한다.

### 구성
- 상단: 페이지 제목 + `새 프로젝트`
- 필터 바
- 프로젝트 카드 또는 테이블

### 각 항목에 보여야 할 정보
- 프로젝트명
- 발주처
- 상태
- 문서 수
- 요구사항 수
- 최근 수정일
- 누락 위험 여부

### 상태 처리
- empty
- loading
- error
- 권한 없음

---

## 6.4 프로젝트 상세 / 분석 대시보드

### 목적
프로젝트 단위 분석 현황을 한 눈에 파악하게 한다.

### 구성
- 프로젝트 개요 카드
- 문서 현황
- 분석 진행 상태
- 요구사항 요약
- 체크리스트 진입 CTA

### 주요 지표
- 총 문서 수
- 분석 완료 문서 수
- 총 요구사항 수
- 검토 필요 수
- 누락 위험 수
- 질의 필요 수

---

## 6.5 문서 페이지

### 목적
프로젝트 내 문서를 관리하고 파싱 상태를 확인하게 한다.

### 구성
- 좌측: 문서 목록
- 우측: 문서 상세 / 업로드 영역

### 표시 정보
- 문서명
- 문서 유형
- 분석 상태
- page_count
- 최근 이력
- 파싱 중 상태 배지

### 업로드 UX 원칙
- 드래그앤드롭 우선
- 업로드 버튼 보조 제공
- 문서 유형 선택을 업로드 근처에 배치

---

## 6.6 요구사항 목록

### 목적
요구사항을 필터/정렬/검색 기반으로 빠르게 검토하게 한다.

### 구성
- 요약 카드
- 필터 바
- 요구사항 테이블

### 핵심 컬럼
- requirementCode
- 제목
- 분류
- analysisStatus
- reviewStatus
- factLevel
- 문서명
- 근거 수

---

## 6.7 요구사항 상세

### 핵심 구조
- 좌: PDF / 원문 근거
- 중: 구조화 분석 정보
- 우: 사람 검토 / 승인 / 코멘트 / 이력

### 중앙 분석 영역 필드 예시
- factSummary
- interpretationSummary
- intentSummary
- proposalPoint
- implementationApproach
- expectedDeliverables
- differentiationPoint
- riskNote
- queryNeededFlag

### 우측 검토 영역 필드 예시
- reviewStatus
- reviewComment
- reviewer
- reviewedAt
- 승인 / 보류 / 검토중 / 수정필요 액션

### 원칙
- AI 분석 영역과 사람 검토 영역의 배경/라벨/배지를 다르게 처리
- 근거 클릭 → PDF 위치 이동 흐름을 가장 중요하게 다룸

---

## 6.8 체크리스트

### 목적
제출/평가/증빙 체크 항목을 상태 중심으로 관리하게 한다.

### 표시 항목
- itemCode
- itemText
- mandatory 여부
- currentStatus
- riskLevel
- riskNote
- linkedRequirement
- sourceExcerpt
- 담당자

### 구성
- 요약 카드
- 필터 바
- 체크리스트 테이블
- 선택 시 상세 패널 또는 인라인 PDF 패널

---

## 6.9 관리자 화면

### 목적
권한, 운영, 통계, 기준정보를 관리하게 한다.

### 원칙
- 일반 사용자 화면보다 더 절제된 디자인
- 장식보다 관리 효율 우선
- 위험 액션은 명확한 Danger 스타일 적용

---

## 7. 상태/정보 구분 규칙

### 7.1 AI 분석 vs 사람 검토
반드시 시각적으로 구분한다.

권장 방식:
- AI 분석: blue/neutral 계열 라벨, `AI 분석`
- 사람 검토: green/orange/gray 계열 라벨, `검토 결과`

금지:
- 같은 카드/박스 안에 섞어 쓰기
- 어떤 정보가 AI인지 사람인지 구분이 안 되는 화면

### 7.2 사실 / 추론 / 검토 필요
권장 표현:
- 사실: 진한 텍스트 + neutral/blue badge
- 추론: amber 계열 badge
- 검토 필요: orange badge

---

## 8. 접근성 / 사용성 원칙

- 색만으로 의미 전달 금지
- 버튼/링크/배지 텍스트 명확히 표기
- 폰트 크기 12px 이하 남발 금지
- 대비 부족한 회색 텍스트 금지
- 빈 상태에는 다음 액션 CTA 제공
- 에러 상태에는 재시도 버튼 제공

---

## 9. 절대 하지 말 것

- 카드 천지로 구성하기
- 과한 파란색 사용
- 과한 border radius
- 그림자 과다 사용
- 작은 글씨 남발
- 상태를 색만으로만 구분
- 긴 설명을 모달에 가둬놓기

## 11. Claude Code 작업 규칙

Claude Code는 화면 수정 시 반드시 아래 규칙을 따른다.

1. **기존 기능을 깨지 않는다.**
2. per-page 임시 스타일을 남발하지 않는다.
3. 가능하면 공통 컴포넌트/토큰으로 정리한다.
4. 소비자 앱처럼 재디자인하지 않는다.
5. 상태 배지, 테이블, 필터 바 중심 구조를 유지한다.
6. 요구사항 상세/체크리스트의 정보 구조는 유지한다.
7. AI 분석과 사람 검토를 절대 섞지 않는다.
8. 큰 리팩토링보다 점진적 UI 정리를 우선한다.
9. empty/loading/error/권한없음 상태를 항상 고려한다.
10. 문구는 짧고 명확하게 유지한다.

## 14. 최종 원칙

BidOps UI의 목표는 **멋진 화면**이 아니라 **신뢰할 수 있는 업무 처리 화면**이다.

화면은 항상 아래 질문에 답할 수 있어야 한다.
- 지금 내가 보고 있는 정보는 무엇인가?
- 이 정보는 AI 분석 결과인가, 사람 검토 결과인가?
- 원문 근거는 어디에 있는가?
- 다음 액션은 무엇인가?

이 질문에 답하지 못하는 UI는 다시 설계한다.
