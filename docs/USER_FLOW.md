# BidOps_USER_FLOW (Updated)

## 1. 문서 목적
이 문서는 BidOps MVP 사용자의 핵심 업무 흐름을 정의한다. 목적은 제품 기획, 화면 설계, API 설계, 개발 구현이 같은 사용자 시나리오를 기준으로 움직이도록 하는 것이다. 본 문서는 상세 와이어프레임 전 단계의 **사용자 흐름 기준서**다.

---

## 2. 사용자 역할 정의
### 2.1 OWNER
- 프로젝트 생성 및 관리
- 멤버 초대/권한 부여
- 분석 실행/승인

### 2.2 EDITOR
- 문서 업로드
- 요구사항 검토/수정
- 체크리스트 조치

### 2.3 REVIEWER
- 분석 결과 검토
- 승인/보류/수정요청

### 2.4 VIEWER
- 조회 전용

---

## 3. 사용자 흐름 설계 원칙
1. 사용자는 “채팅”보다 **업무 흐름**으로 제품을 사용한다.
2. 첫 핵심 가치는 “RFP 업로드 후 분석 결과를 신뢰성 있게 보는 경험”이다.
3. 모든 주요 단계에는 상태값과 다음 액션이 보여야 한다.
4. 사용자는 항상 원문 근거로 다시 이동할 수 있어야 한다.
5. AI 분석은 자동 제안이지만, 사람 검토가 기본 흐름이다.
6. AI 처리 상태와 사람 검토 상태를 같은 축으로 섞지 않는다.

---

## 4. 전체 MVP 사용자 흐름
```text
로그인
  → 프로젝트 목록
    → 프로젝트 생성
      → 프로젝트 상세
        → RFP 문서 업로드
          → 문서 파싱 / OCR 실행
            → 요구사항 추출 완료
              → RFP 분석 결과 대시보드
                → 요구사항 목록 검토
                  → 요구사항 상세 분석 확인
                    → 분석 확인 / 검토 코멘트 / 승인 / 보류 / 수정요청
                → 체크리스트 검토
                  → 항목별 조치
                → 유사 제안 검색
                  → 참고자료 확인
```

## 5. 핵심 시나리오별 상세 흐름
## 5.1 시나리오 A: 신규 프로젝트 생성과 RFP 분석 시작
### 목표
사용자가 새로운 입찰 건을 등록하고, RFP를 업로드해 분석 파이프라인을 시작한다.

### 선행 조건
- 사용자가 로그인 완료 상태
- 프로젝트 생성 권한 보유

### 흐름
1. 사용자는 프로젝트 목록 화면에 진입한다.
2. “새 프로젝트 생성” 버튼을 누른다.
3. 프로젝트 기본정보를 입력한다.
   - 프로젝트명
   - 발주처명
   - 사업명
   - 공고번호(선택)
   - 설명(선택)
4. 생성 완료 후 프로젝트 상세 화면으로 이동한다.
5. 사용자는 RFP 문서를 업로드한다.
6. 시스템은 문서 유형을 확인하고 업로드를 완료한다.
7. 사용자는 “분석 시작” 액션을 실행한다.
8. 시스템은 OCR/레이아웃 분석 → 요구사항 추출 Job을 생성한다.
9. 프로젝트 상세 화면에서 분석 진행 상태를 노출한다.
10. 분석 완료 후 사용자에게 결과 진입 CTA를 제공한다.

### 예외 상황
- 잘못된 파일 형식 업로드
- 문서가 손상되어 파싱 실패
- OCR 실패 또는 일부 페이지 품질 저하

### 성공 기준
- 프로젝트 생성 후 5분 내 첫 분석 결과 화면 진입 가능

---

## 5.2 시나리오 B: 요구사항 목록 검토
### 목표
사용자가 추출된 요구사항을 카테고리별로 보고, 우선순위와 리스크를 파악한다.

### 흐름
1. 사용자는 분석 결과 대시보드에서 “요구사항 목록 보기”를 클릭한다.
2. 시스템은 요구사항 목록을 테이블 형태로 보여준다.
3. 사용자는 필터를 사용한다.
   - 분류
   - 필수 여부
   - 증빙 필요 여부
   - 분석 상태
   - 검토 상태
   - Fact Level
4. 특정 요구사항을 선택하면 상세 패널 또는 상세 화면으로 이동한다.
5. 사용자는 원문, 해석, 제안 포인트, 리스크, 질의 필요 여부를 확인한다.
6. 사용자는 검토 상태를 변경한다.
   - 승인
   - 수정 필요
   - 보류
7. 시스템은 검토 이력을 저장한다.

### 화면 핵심 포인트
- 테이블 중심
- 상태배지 명확
- 원문 근거 페이지 링크 제공

### 성공 기준
- 사용자가 요구사항 누락 여부를 빠르게 훑을 수 있어야 함
- 상세 진입 후 10초 이내에 원문 근거 위치 확인 가능해야 함

---

## 5.3 시나리오 C: 요구사항 상세 분석 검토
### 목표
사용자가 단일 요구사항을 실무 수준으로 검토하고, 제안서 반영 포인트를 얻는다.

### 흐름
1. 사용자는 요구사항 상세 화면으로 진입한다.
2. 시스템은 다음 정보를 보여준다.
   - 요구사항 코드
   - 원문
   - 문서명
   - 분류
   - 필수 여부
   - 증빙 필요 여부
   - 분석 상태
   - 검토 상태
   - Fact Level
3. 화면은 최소 3개 영역으로 나뉜다.
   - 분석 영역: 사실 요약, 해석 요약, 발주처 의도, 제안 포인트, 구현 방향, 산출물, 차별화 포인트, 리스크, 질의 필요 여부
   - 검토 영역: review 상태, review 코멘트, 검토자, 검토 시각
   - 근거 영역: 원문 발췌, 페이지 번호, anchor, PDF 원문 연결
4. 사용자는 분석 결과를 수정하거나 검토 코멘트를 남긴다.
5. 필요 시 검토 상태를 변경한다.
6. 시스템은 AI 분석 수정 이력과 사람 검토 이력을 구분해 저장한다.

### 예외 상황
- AI 해석이 과도하거나 근거 부족
- 서로 상충하는 요구사항 발견
- 원문이 애매해 질의 필요

### 성공 기준
- 단일 요구사항에서 “바로 제안서에 반영할 수 있는 포인트”가 보여야 함

---

## 5.4 시나리오 D: 체크리스트 검토 및 누락 방지
### 목표
사용자가 제출서류/증빙/평가항목 반영 누락을 발견하고 조치한다.

### 흐름
1. 사용자는 분석 결과 대시보드에서 “체크리스트” 탭으로 이동한다.
2. 시스템은 체크리스트를 유형별로 보여준다.
   - 제출서류
   - 증빙자료
   - 평가항목 반영
3. 각 항목은 **상태**와 **위험도**를 가진다.
4. 사용자는 항목별로 상태를 갱신한다.
   - 미착수
   - 진행중
   - 완료
   - 차단됨
5. 위험도는 별도 표시된다.
   - 없음
   - 낮음
   - 보통
   - 높음
6. 필요 시 관련 요구사항이나 원문 근거로 이동한다.
7. 시스템은 완료율과 고위험 항목 수를 대시보드에 반영한다.

### 성공 기준
- 사용자가 “무엇이 빠졌는지” 한눈에 볼 수 있어야 함

---

## 5.5 시나리오 E: 유사 제안 검색
### 목표
사용자가 과거 유사 제안이나 참고 문구를 찾아 현재 프로젝트에 활용한다.

### 흐름
1. 사용자는 요구사항 상세 또는 별도 검색 화면에서 “유사 제안 검색”을 클릭한다.
2. 시스템은 같은 분류/유사 의미 기반의 검색 결과를 보여준다.
3. 사용자는 결과를 확인한다.
   - 유사 요구사항
   - 실적 설명 문구
   - 산출물 구조
4. 사용자는 필요한 내용을 참조한다.
5. 이후 제안서 작성 단계에서 활용한다.

### 주의
- 검색 결과는 참고용이며 자동 삽입보다 “검토 후 활용” 중심으로 설계한다.

## 6. 화면 전이 기준
### 6.1 프로젝트 목록 → 프로젝트 상세
트리거:
- 프로젝트 카드 클릭
- 신규 프로젝트 생성 완료

### 6.2 프로젝트 상세 → 분석 결과 대시보드
트리거:
- 분석 Job 완료
- 사용자 CTA 클릭

### 6.3 분석 결과 대시보드 → 요구사항 목록
트리거:
- 요구사항 섹션 클릭
- 요약 카드 클릭

### 6.4 요구사항 목록 → 요구사항 상세
트리거:
- 테이블 Row 클릭
- 상세보기 액션 클릭

### 6.5 요구사항 상세 ↔ 체크리스트
트리거:
- 연결된 체크리스트 항목 클릭
- 관련 요구사항 링크 클릭

## 8. 사용자별 핵심 가치 흐름
### 8.1 OWNER
로그인 → 프로젝트 생성 → RFP 업로드 → 분석 실행 → 결과 검토 지시 → 승인 관리

### 8.2 EDITOR
프로젝트 진입 → 요구사항 목록 검토 → 상세 분석 수정 → 체크리스트 조치 → 유사 제안 검색

### 8.3 REVIEWER
검토 대상 확인 → 요구사항 상세 근거 확인 → AI 분석 확인 → review 조회 → 승인/보류/수정요청 → 리스크 항목 검토

---

## 9. 비정상 흐름 및 대응
### 9.1 OCR 실패
- 사용자에게 실패 메시지 제공
- 실패 페이지 표시
- 재시도 기능 제공

### 9.2 요구사항 추출 품질 낮음
- “검토 필요” 배지 강조
- 원문 중심 화면 제공
- 재분석 요청 가능

### 9.3 문서 중복 업로드
- 버전 업로드로 안내
- 기존 버전과 비교 이력 유지

### 9.4 권한 부족
- 읽기 전용 화면 전환 또는 접근 제한 메시지 노출

---

## 10. UX 성공 기준
1. 사용자는 업로드 후 분석이 어디까지 진행됐는지 항상 알 수 있어야 한다.
2. 사용자는 요구사항 하나를 볼 때 원문 근거, 해석, 제안 포인트를 동시에 확인할 수 있어야 한다.
3. 사용자는 체크리스트에서 누락 위험을 빠르게 인지할 수 있어야 한다.
4. 사용자는 AI 결과를 맹신하지 않고 검토 가능한 구조를 경험해야 한다.
5. 사용자는 AI 분석 정보와 사람 검토 정보를 혼동하지 않아야 한다.
