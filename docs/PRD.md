## 1. 제품 개요

**BidOps**는 공공/민간 입찰·제안 업무에서 RFP를 구조적으로 분석하고, 요구사항을 항목화하며, 누락 위험을 줄이고, 제안서 반영과 수주 운영을 지원하는 **근거 기반 문서 운영 AI OS**다.

이 제품의 핵심은 “제안서를 자동으로 써주는 도구”가 아니라, **신뢰 가능한 분석 결과를 바탕으로 사람이 더 빠르고 정확하게 제안 업무를 수행하도록 돕는 시스템**이라는 점이다.

### 1.1 핵심 가치
1. RFP 해석 시간 단축
2. 요구사항 누락 방지
3. 원문 근거 기반 분석 제공
4. 제안 품질 표준화
5. 과거 제안 자산 재사용 효율화
6. 검토·승인 기반 수주 운영 체계화

### 1.2 제품 철학
- 생성보다 검증 가능한 분석을 우선한다.
- AI 해석과 사실 추출을 분리한다.
- 모든 결과는 가능한 한 원문 근거와 연결한다.
- 최종 제출 전에는 반드시 사람 검토·승인을 거친다.
- 화려한 자동화보다 실무 신뢰도를 우선한다.

## 2. 목표와 비목표

### 2.1 목표
- 사용자가 RFP PDF를 업로드하면 요구사항을 구조화하여 검토 가능한 수준으로 제공한다.
- 필수 제출물, 증빙, 실적, 인력, 일정 관련 누락 위험을 조기에 식별한다.
- 제안 PM이 프로젝트 단위로 문서와 분석 결과를 관리할 수 있게 한다.
- 요구사항별로 제안서 반영 포인트와 구현 방향의 초안을 제공한다.
- 초기 파일럿에서 “참고용”이 아니라 “검토용”으로 사용할 수 있는 수준을 만든다.

### 2.2 비목표
- 버튼 한 번으로 제안서 전체를 자동 작성하는 기능
- 완전 무인 제출 판단
- 고급 발표자료 자동 디자인
- 복잡한 협력사/그룹웨어 대규모 연동
- 견적/원가/가격 산정 자동화

## 3. 대상 사용자

### 3.1 1차 사용자
- SI 업체 제안 PM
- 중소 IT기업 사업기획/영업 담당자
- 교육/플랫폼 구축 사업 참여 업체 담당자
- 공공사업 참여 컨설팅/수행사 실무자

### 3.2 사용자 페르소나

#### A. 제안 PM
- 사업공고를 받자마자 RFP를 빠르게 파악해야 한다.
- 누락 없는 체크리스트와 우선순위가 필요하다.
- 실무자에게 역할을 배분해야 한다.

#### B. 작성 실무자
- 요구사항을 실제 제안서 내용과 산출물로 풀어야 한다.
- 원문 확인 시간이 오래 걸린다.
- 유사 사업 자산을 빠르게 찾고 싶다.

#### C. 임원/대표
- 제출 리스크를 낮추고 싶다.
- 제안 역량을 조직 차원에서 표준화하고 싶다.
- 담당자 경험 의존도를 줄이고 싶다.

## 4. 문제 정의

### 4.1 현재 문제
- RFP 분량이 많고 부속 자료가 많아 핵심 요구사항 누락 위험이 높다.
- 해석 품질이 사람 개인 역량에 의존한다.
- 질의응답, 증빙, 실적, 인력, 평가기준 관리가 분산된다.
- 기존 AI 요약은 근거·정확성·검토 가능성이 부족하다.
- 과거 제안서 재사용이 비효율적이다.

### 4.2 사용자가 진짜 원하는 것
- 이 RFP를 믿고 해석해도 되는가?
- 무엇을 반드시 넣어야 하는가?
- 어디가 리스크인가?
- 제안서에는 어떻게 풀어야 하는가?
- 누락은 없는가?

## 5. MVP 범위

### 5.1 MVP 목표
RFP 업로드부터 분석 결과 검토까지의 핵심 흐름을 구현한다.

### 5.2 MVP 포함 범위
1. 로그인
2. 프로젝트 생성/목록 관리
3. 문서 업로드(RFP PDF 중심)
4. 문서 구조 추출(OCR/레이아웃)
5. 요구사항 자동 분해
6. 요구사항 목록/상세 분석
7. 필수 제출물·증빙 체크리스트
8. 분석 근거 페이지 연결
9. 검토/승인 상태 관리
10. 기초 수준의 유사 제안 검색

### 5.3 MVP 제외 범위
- 자동 PPT 디자인 생성
- 복잡한 다중 에이전트 orchestration
- 고객사별 고급 템플릿 엔진
- 외부 그룹웨어/전자결재 통합
- 자동 견적 및 원가 산정
- 모바일 앱 전용 UX

## 6. 핵심 사용자 흐름

### Flow 1. 프로젝트 생성 및 문서 업로드
1. 사용자가 로그인한다.
2. 새 프로젝트를 생성한다.
3. RFP PDF를 업로드한다.
4. 시스템은 문서 구조 추출 작업을 시작한다.
5. 사용자는 분석 상태를 확인한다.

### Flow 2. RFP 분석 검토
1. 문서 구조 추출이 완료되면 요구사항 분해가 수행된다.
2. 사용자는 요구사항 목록 화면에서 전체 항목을 확인한다.
3. 사용자는 각 요구사항 상세 화면에서 원문, 의도, 반영 포인트, 리스크를 검토한다.
4. 애매하거나 틀린 항목은 수정/보류 처리한다.

### Flow 3. 누락 체크 및 반영
1. 사용자는 체크리스트 화면에서 필수 제출물/증빙/실적/인력 관련 누락 가능 항목을 확인한다.
2. 프로젝트 상태를 보며 우선순위를 정한다.
3. 요구사항을 제안서 작성 업무로 배분한다.

### Flow 4. 자산 재사용
1. 사용자는 유사 요구사항 또는 과거 제안 자산을 검색한다.
2. 실적 문구, 산출물 구조, 작성 포인트를 참고한다.
3. 신규 제안서에 재활용한다.

## 8. 신뢰성 요구사항

### 8.1 기본 원칙
- 분석 결과는 가능한 한 원문 근거를 명시해야 한다.
- 사실 추출과 추론/해석을 구분해 표기해야 한다.
- 신뢰도 낮은 항목은 “검토 필요”로 표시해야 한다.
- 사람 승인 없는 자동 제출 판단을 제공하지 않는다.

### 8.2 요구사항 분석 포맷
모든 요구사항은 최소 아래 구조를 가진다.
1. 요구사항 ID
2. 원문
3. 문서/페이지/조항 위치
4. 분류
5. 필수 여부
6. 증빙 필요 여부
7. 발주처 의도
8. 제안서 반영 포인트
9. 구현 방향
10. 필요 산출물
11. 차별화 포인트
12. 리스크
13. 질의 필요 여부
14. 확정/추론/검토필요 구분
15. 최종 검토 상태

### 8.3 품질 기준
- 원문 근거 없는 확정 표현 금지
- 누락 가능 항목은 이유와 함께 표시
- 조항 충돌 시 경고 표시
- 동일 의미 중복 요구사항은 연결 또는 병합 제안
- 사용자가 수정한 결과를 우선 저장

## 9. 비기능 요구사항

### 9.1 성능
- 일반적인 RFP 업로드 후 1차 분석 시작까지 대기 상태를 명확히 표시해야 한다.
- 분석 작업은 비동기 처리해야 한다.
- 사용자는 작업 중에도 프로젝트/기존 결과를 조회할 수 있어야 한다.

### 9.2 보안
- 프로젝트별 접근 권한을 분리한다.
- 문서 원본과 분석 결과는 저장소에서 보호되어야 한다.
- 분석/수정/승인 이력은 감사 로그 형태로 남긴다.

### 9.3 사용성
- 업무형 B2B UI를 유지한다.
- 표/필터/상태 배지 중심으로 빠르게 스캔 가능해야 한다.
- 근거 원문으로 이동하는 액션이 명확해야 한다.

### 9.4 확장성
- 추후 다문서 입력, 질의응답 이력, 고급 검색, 발표자료 전환, 협업 워크플로우 확장이 가능해야 한다.
- 분석 엔진과 앱 서버를 분리 가능하도록 설계한다.

## 10. 화면 범위

### 10.1 화면 목록
1. 로그인
2. 프로젝트 목록
3. 프로젝트 상세
4. 문서 업로드
5. RFP 분석 결과 대시보드
6. 요구사항 목록
7. 요구사항 상세 분석
8. 누락/증빙 체크리스트
9. 유사 제안 검색
10. 관리자/설정

### 10.2 1차 핵심 화면 상세

#### Screen A. 프로젝트 목록
- 목적: 전체 프로젝트 진행 상태를 한눈에 파악
- 핵심 요소: 검색, 상태 필터, 프로젝트 카드/테이블, 최근 업데이트, 마감일
- 주요 액션: 새 프로젝트 생성, 프로젝트 상세 이동
- 상태값: 분석전, 분석중, 검토필요, 승인완료

#### Screen B. 프로젝트 상세 / 문서 업로드
- 목적: 프로젝트 기본정보와 업로드 문서 관리
- 핵심 요소: 프로젝트 개요, 문서 리스트, 업로드 버튼, 문서 상태, 최근 분석 이력
- 주요 액션: 문서 업로드, 분석 재실행, 문서 버전 확인

#### Screen C. RFP 분석 결과
- 목적: 분석된 요구사항과 체크리스트를 중심으로 핵심 판단 지원
- 핵심 요소: 분석 요약 카드, 요구사항 수, 필수 제출물 수, 누락 위험, 질의 필요 항목, 탭(요구사항/체크리스트/검색)
- 주요 액션: 요구사항 상세 열기, 상태 변경, 수정 저장

#### Screen D. 요구사항 상세 분석
- 목적: 개별 요구사항을 깊게 검토하고 수정
- 핵심 요소: 원문, 근거 위치, 분석 포맷, 리스크, 산출물, 검토 상태
- 주요 액션: 수정, 승인, 보류, 질의 필요 표시

#### Screen E. 누락/증빙 체크리스트
- 목적: 제출 리스크를 빠르게 확인
- 핵심 요소: 항목명, 유형, 누락 가능성, 근거, 조치 상태, 담당자

## 14. 리스크 및 대응

| 리스크 | 설명 | 대응 방향 |
|---|---|---|
| OCR 품질 불안정 | 스캔본·표·양식 품질 문제 | 문서 구조화 엔진 우선 검증, 문서 유형 제한 |

| 과도한 자동화 기대 | 사용자가 자동 작성기로 오해 | 제품 메시지를 “근거 기반 분석”으로 고정 |
| 범위 과대 | 초기부터 기능이 너무 많아짐 | MVP 범위 엄격 통제 |
| 신뢰도 하락 | 근거 없는 해석 또는 누락 발생 | 원문 연결, 검토 상태, 승인 흐름 필수화 |
