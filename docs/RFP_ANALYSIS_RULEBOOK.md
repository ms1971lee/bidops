# BidOps RFP_ANALYSIS_RULEBOOK

## 1. 문서 목적
이 문서는 BidOps에서 RFP(제안요청서)를 분석할 때 따라야 하는 **고정 분석 기준**을 정의한다.
목표는 AI가 그럴듯하게 요약하는 것이 아니라, **원문 근거를 유지한 상태에서 요구사항을 구조적으로 분해하고, 제안서 반영 포인트와 리스크를 검토 가능한 형태로 제시하는 것**이다.

---

## 2. 핵심 원칙
1. **원문 우선**
   - 모든 분석은 원문 조항에서 출발한다.
   - 분석 결과는 가능하면 문서명, 페이지, 조항 번호와 연결한다.

2. **사실과 해석 분리**
   - 사실 추출: 원문에 명시된 내용
   - 추론 정보: 원문을 바탕으로 도출한 해석
   - 검토 필요 정보: 애매하거나 충돌 가능성이 있는 항목

3. **요약보다 구조화**
   - 자유서술형 장문보다 구조화된 항목을 우선한다.
   - 각 요구사항은 동일한 템플릿으로 분석한다.

4. **누락 방지가 핵심**
   - 이 시스템의 1차 가치는 문장을 예쁘게 쓰는 것이 아니라, 필수 조건/제출물/증빙/평가 기준을 빠뜨리지 않는 데 있다.

5. **AI 단독 확정 금지**
   - AI는 1차 분석자다.
   - 최종 확정은 실무자 검토 및 승인 흐름을 거친다.

---

## 3. 분석 대상 범위
분석 대상에는 아래 항목이 포함된다.

- 본문 요구사항
- 사업 개요
- 추진 배경 및 목적
- 과업 범위
- 기능 요구사항
- 비기능 요구사항
- 보안 요구사항
- 품질/성능 요구사항
- 투입인력 조건
- 실적 조건
- 제출 서류 및 증빙
- 제안서 작성 지침
- 평가 기준 및 배점
- 발표평가 관련 조건
- 사업기간/일정
- 산출물 목록
- 유지보수/하자보수/교육 조건
- 별첨 양식, 표, 체크리스트, 첨부파일

---

## 4. 분석 단위 정의
RFP는 문단 단위 요약으로 끝내지 않는다.
분석의 최소 단위는 **요구사항 또는 검토 포인트 1개**다.

예시:
- “최근 3년 이내 유사 사업 실적 2건 이상”
- “PM은 유사 사업 경험 5년 이상”
- “발표자료는 15장 이내”
- “개인정보 암호화 적용”
- “개발 완료 후 사용자 교육 2회 이상”

즉, 한 페이지에 요구사항이 여러 개 있으면 각각 분리해 저장한다.

---

## 5. 요구사항 분류 체계
각 요구사항은 아래 분류 중 1개 이상으로 태깅한다.

### 5.1 기본 분류
- 사업개요
- 추진배경
- 추진목표
- 과업범위
- 기능요구사항
- 비기능요구사항
- 성능요구사항
- 보안요구사항
- 품질관리
- 테스트/검증
- 데이터/연계
- UI/UX
- 인프라/운영환경
- 인력요건
- 실적요건
- 일정/기간
- 산출물
- 제출서류
- 제안서작성지침
- 평가기준
- 발표평가
- 유지보수
- 교육/전환
- 법률/규정준수
- 기타

### 5.2 성격 분류
- 필수
- 권장
- 선택
- 확인필요
- 질의필요

---

## 6. 고정 출력 포맷
모든 요구사항 분석은 아래 형식을 따른다.

### 6.1 필수 필드
1. 요구사항 ID
2. 문서명
3. 페이지 번호
4. 조항/표/부속 위치
5. 요구사항 원문
6. 요구사항 요약
7. 분류
8. 필수 여부
9. 증빙 필요 여부
10. 관련 제출물/산출물
11. 발주처 의도
12. 제안서 반영 포인트
13. 구현 방향
14. 차별화 포인트
15. 리스크/주의사항
16. 질의 필요 여부
17. 확정/추론/검토 필요 구분
18. 신뢰도
19. 검토 상태

### 6.2 신뢰도 기준
- 높음: 원문이 명확하고 해석 여지가 거의 없음
- 중간: 원문은 있으나 표현이 다소 추상적임
- 낮음: 문맥 의존적이거나 모호함, 부속문서 확인 필요

---

## 7. 각 필드 작성 기준

### 7.1 요구사항 원문
- 원문을 축약하지 말고 핵심 표현을 최대한 그대로 남긴다.
- 표 항목이면 표의 행/열 의미를 함께 적는다.

### 7.2 요구사항 요약
- 1~2문장 내로 명료하게 정리한다.
- 원문을 왜곡하지 않는다.

### 7.3 발주처 의도
아래 질문에 답하는 방식으로 작성한다.
- 발주처가 왜 이 항목을 넣었는가?
- 무엇을 중시하는가?
- 평가 시 어떤 관점에서 볼 가능성이 큰가?

### 7.4 제안서 반영 포인트
아래 중 해당되는 것을 구체화한다.
- 제안서 어느 장/절에 반드시 반영해야 하는가
- 어떤 표현/자료/근거가 필요한가
- 어떤 그림/표/프로세스/일정표로 풀면 좋은가

### 7.5 구현 방향
아래 요소를 중심으로 작성한다.
- 시스템/기능/운영 관점의 대응 방안
- 연계 구조, 데이터 흐름, 운영 절차
- 보안/품질/성능 대응

### 7.6 차별화 포인트
단순히 “우수하게 수행” 같은 문장을 쓰지 않는다.
반드시 아래 중 하나 이상으로 구체화한다.
- 방법론 차별화
- 수행조직/거버넌스 차별화
- 품질관리 체계 차별화
- 리스크 대응 체계 차별화
- 자동화/표준화/확장성 차별화

### 7.7 리스크/주의사항
반드시 실무 리스크 관점에서 적는다.
예시:
- 해석이 애매해 질의 필요
- 증빙 누락 시 실격 가능
- 제안서 분량 제한과 충돌 가능
- 타 조항과 일정/범위 충돌 가능
- 별첨 양식 확인 필요

### 7.8 질의 필요 여부
아래 중 하나로 표시한다.
- 질의 불필요
- 질의 권장
- 질의 필요(모호/충돌/기준 불명확)

---

## 8. 확정/추론/검토 필요 구분 기준

### 8.1 확정 정보
원문에 직접 명시되어 있고 해석 여지가 거의 없는 항목
예:
- 제출 마감일
- 제안서 부수
- 사업 기간
- 실적 기준 수치

### 8.2 추론 정보
원문을 토대로 합리적으로 도출했으나, 발주처가 명시적으로 쓰지 않은 항목
예:
- “운영 안정성 중시”
- “유지보수 대응 체계를 평가할 가능성 높음”

### 8.3 검토 필요 정보
모호하거나, 여러 조항과 충돌하거나, 부속문서 확인이 필요한 항목
예:
- 본문과 별첨 양식 간 기준 불일치
- 인력 경력 기준이 불분명
- 평가 방식이 정성인지 정량인지 애매함

---

## 9. 분석 절차 표준

### Step 1. 문서 구조 파악
- 문서명, 버전, 페이지 수 확인
- 목차/장/절/부속문서 구조 파악
- 표, 체크리스트, 별첨 양식 위치 식별

### Step 2. 원문 추출
- 페이지 단위 텍스트 추출
- 표/목록/체크박스 포함 구조 추출
- 조항 번호와 문단 범위 연결

### Step 3. 요구사항 분해
- 복수 요구가 섞인 문장은 단일 요구사항으로 분리
- 하나의 표에서 여러 조건이 있으면 개별 분리

### Step 4. 분류 및 태깅
- 요구사항 유형 분류
- 필수/권장/선택/질의필요 태깅

### Step 5. 구조화 분석
- 고정 출력 포맷에 맞춰 분석 작성
- 근거 페이지와 조항 연결

### Step 6. 누락 탐지
아래 누락 여부를 확인한다.
- 제출서류 누락
- 실적/인력 조건 누락
- 평가 기준 반영 누락
- 일정/범위/산출물 누락
- 발표 자료 관련 조건 누락

### Step 7. 충돌/모호성 점검
- 본문과 별첨 간 상충 여부
- 일정과 범위 충돌 여부
- 필수요건과 평가요건 중복/불일치 여부
- 모호 표현 존재 여부

### Step 8. 사람 검토 대기
- 검토자 확인 전에는 자동 확정 금지
- 수정 이력과 승인 상태 기록

## 10. 누락 탐지 체크리스트 기준
아래는 시스템이 기본적으로 점검해야 하는 영역이다.

### 10.1 제출/형식
- 제출 마감일/시간
- 제출처
- 파일 형식
- 인쇄/제본 요건
- 발표자료 분량
- 제안서 분량 제한
- 서명/날인 필요 여부

### 10.2 실적/인력
- 유사 실적 기준
- 기간 기준
- 금액 기준
- 역할 기준
- 자격증/경력증명 필요 여부

### 10.3 기능/기술
- 필수 기능
- 연계 대상 시스템
- 데이터 이관 조건
- 성능 기준
- 보안 기준

### 10.4 사업관리
- 일정표 필요 여부
- 보고 체계
- 회의체 운영
- 품질관리 계획
- 위험관리 방안

### 10.5 산출물/운영
- 산출물 종류
- 교육 자료
- 매뉴얼
- 테스트 결과서
- 유지보수 계획
- 하자보수 조건

## 11. 좋은 분석과 나쁜 분석 예시

### 11.1 나쁜 분석 예시
요구사항: “개인정보 보호에 유의해야 한다.”
분석: “보안을 잘 해야 한다.”

문제점:
- 원문 근거가 약함
- 실무 대응이 없음
- 제안 포인트가 없음
- 리스크 판단이 없음

### 11.2 좋은 분석 예시
요구사항 원문: “시스템은 개인정보보호법 등 관련 법령을 준수하여 개인정보를 안전하게 처리하여야 한다.”
발주처 의도: 법규 위반 리스크를 방지하고 개인정보 처리의 안전성을 확보하려는 요구사항이다.
제안서 반영 포인트: 보안 아키텍처, 접근권한 통제, 암호화, 접속기록 관리, 개인정보 처리 흐름을 별도 절로 반영해야 한다.
구현 방향: 저장 구간 암호화, 전송 구간 TLS 적용, 권한 기반 접근통제, 로그 감사, 개인정보 마스킹 정책을 명시한다.
리스크: 법규 준수 항목을 추상적으로만 쓰면 평가 시 신뢰도가 낮아질 수 있다.
질의 필요 여부: 질의 불필요.

---

## 12. AI 프롬프트 작성 원칙
분석 AI에 입력하는 프롬프트는 아래 원칙을 따른다.

1. “요약”보다 “구조화 추출”을 지시한다.
2. 반드시 원문 근거와 페이지 번호를 함께 출력하게 한다.
3. 확정/추론/검토 필요를 구분하게 한다.
4. 질의 필요 여부를 별도 판단하게 한다.
5. 근거가 약한 경우 추정으로 표시하게 한다.
6. 자유로운 마케팅 문구 생성을 금지한다.
7. 근거 없는 ‘우수’, ‘최적’, ‘완벽’ 같은 표현을 금지한다.

---

## 13. 사람이 최종 검토할 때 보는 기준
실무 검토자는 아래 항목을 확인한다.

- 원문 누락 없이 분해되었는가
- 조항 위치가 정확한가
- 해석이 과도하지 않은가
- 질의 필요 항목이 잘 드러나는가
- 제안서 반영 포인트가 실무적으로 유효한가
- 리스크가 실제 제출 관점에서 도움이 되는가

---

## 14. 시스템 금지사항
아래와 같은 결과는 금지한다.

- 원문 근거 없는 단정
- 한 페이지 전체를 뭉뚱그려 요약
- 요구사항 여러 개를 한 항목으로 처리
- 실적/인력/제출서류 누락
- 평가 기준과 제안 포인트의 연결 부재
- 추론 정보를 확정 사실처럼 표기
- 사람이 검토해야 할 항목을 자동 확정

---

## 15. 운영 원칙 요약
BidOps의 RFP 분석은 아래 구조를 따른다.

**원문 추출 → 요구사항 분해 → 구조화 분석 → 누락/충돌 점검 → 사람 검토 → 승인**

이 문서는 앞으로 모든 RFP 분석, 프롬프트 설계, 기능 구현, QA 검토의 기준 문서로 사용한다.
