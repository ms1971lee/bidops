# BidOps 프로젝트 기획 초안

## 1. 프로젝트 한 줄 정의
**BidOps**는 공공/민간 입찰·제안 업무에서 제안요청서(RFP)를 구조적으로 분석하고, 요구사항 누락을 방지하며, 제안서 작성과 수주 운영을 지원하는 **근거 기반 문서 운영 AI OS**다.

---

## 2. 왜 이 프로젝트를 하는가
기존 생성형 AI 기반 문서 도구는 문장을 그럴듯하게 만들 수는 있지만, 입찰/제안 업무에서 가장 중요한 **정확성, 누락 방지, 근거 제시, 검토 가능성**이 약하다.

입찰/제안 실무자는 다음과 같은 문제를 겪는다.

- RFP 분량이 길고 부속 문서가 많아 핵심 요구사항 누락 위험이 크다.
- 제안서 작성이 사람 경험에 과도하게 의존한다.
- 질의응답, 증빙, 실적, 인력, 평가기준 관리가 분산되어 있다.
- AI 요약은 많지만, **실무에 제출 가능한 수준의 신뢰성**은 부족하다.

BidOps는 이 문제를 해결하기 위해, 단순 제안서 작성기가 아니라 **수주 운영 시스템**으로 설계한다.

---

## 3. 제품 철학
### 3.1 핵심 철학
이 제품의 본질은 **생성**이 아니라 **검증 가능한 분석**이다.

### 3.2 반드시 지켜야 할 원칙
1. 모든 분석 결과는 가능한 한 **원문 근거**와 연결되어야 한다.
2. AI 해석과 사실 추출을 구분한다.
3. 결과는 자유서술보다 **구조화된 항목**으로 제공한다.
4. 누락 탐지, 충돌 탐지, 모호성 탐지가 핵심 가치다.
5. 최종 제출은 반드시 사람이 검토/승인하는 흐름을 가진다.

### 3.3 하지 않을 것
- 제안서를 버튼 한 번으로 자동 완성하는 제품으로 포지셔닝하지 않는다.
- 근거 없는 추론을 확정 정보처럼 보이게 하지 않는다.
- 화려한 UI를 위해 분석 정확도를 희생하지 않는다.

## 4. 목표 고객
### 4.1 1차 타깃
- SI 업체 제안 PM
- 중소 IT기업 사업기획/영업 담당자
- 교육/플랫폼 구축 사업 참여 업체
- 공공사업 참여 컨설팅/수행사

### 4.2 2차 타깃
- 중견기업 제안조직
- 입찰 지원 전문 조직
- PMO/사업관리 조직

### 4.3 사용자 페르소나
#### 페르소나 A: 제안 PM
- 사업공고 확인 후 RFP를 빠르게 해석해야 함
- 누락 없이 제안 방향을 잡아야 함
- 실무자/협력사에게 역할 배분해야 함

#### 페르소나 B: 작성 실무자
- 각 요구사항을 실제 제안서 문장과 산출물로 풀어야 함
- 원문을 반복 확인하는 시간이 많이 듦

#### 페르소나 C: 대표/임원
- 제출 리스크를 낮추고, 조직의 제안 역량을 표준화하고 싶어 함

---

## 5. 핵심 문제 정의
### 5.1 현재 문제
- 요구사항 해석이 사람 역량에 과도하게 의존
- 필수 제출물/증빙 누락 위험 상존
- 평가항목과 제안내용 간 정합성 부족
- 질의응답/버전/문서 이력 관리 어려움
- 유사 제안 재사용이 비효율적

### 5.2 고객이 진짜 원하는 것
- “이 RFP를 믿고 해석해도 되는가?”
- “무엇을 반드시 넣어야 하는가?”
- “어디가 리스크인가?”
- “이걸 제안서에 어떻게 풀어야 하는가?”

---

## 6. 가치 제안
BidOps는 다음 가치를 제공한다.

1. **RFP 분석 시간 단축**
2. **누락 위험 감소**
3. **제안 품질 표준화**
4. **근거 기반 해석 제공**
5. **과거 제안서/실적 재사용 효율화**
6. **수주 운영 흐름 체계화**

---

## 7. 제품 포지셔닝
### 7.1 무엇인가
- RFP 분석 엔진
- 요구사항 구조화 엔진
- 누락/증빙 체크 엔진
- 제안 운영 협업 시스템

### 7.2 무엇이 아닌가
- 단순 AI 챗봇
- 단순 제안서 자동작성기
- 일반 문서 요약툴

### 7.3 포지셔닝 문장
**BidOps는 공공/민간 제안 업무에서 요구사항을 근거 기반으로 분석하고, 누락을 방지하며, 제안서 반영과 수주 운영을 체계화하는 문서 운영 AI OS다.**

---

## 8. MVP 범위
### 8.1 MVP 목표
사용자가 RFP PDF를 업로드하면, 시스템이 다음을 제공하는 수준까지 구현한다.

- 문서 업로드 및 프로젝트 생성
- RFP 구조 추출
- 요구사항 자동 분해
- 요구사항별 구조화 분석
- 필수 제출물/증빙 체크리스트 생성
- 누락 가능 항목 표시
- 분석 결과 검토/수정
- 유사 제안 검색(기초)

### 8.2 MVP에서 반드시 포함할 기능
1. 로그인/프로젝트 생성
2. RFP PDF 업로드
3. 문서 파싱 및 구조화
4. 요구사항 리스트 화면
5. 요구사항 상세 분석 화면
6. 누락/필수증빙 체크리스트
7. 분석 근거 페이지 연결
8. 분석 결과 승인 상태 관리

### 8.3 MVP에서 제외할 기능
- 완전 자동 PPT 생성
- 고급 협업 워크플로우
- 복잡한 다중 에이전트
- 고객사별 고급 템플릿 엔진
- 가격 산정/견적 자동화
- 외부 그룹웨어 대규모 연동

## 9. 핵심 기능 상세 정의
### 9.1 프로젝트/문서 관리
- 프로젝트 생성
- 문서 업로드
- 문서 유형 지정
- 버전 관리
- 상태값 관리

### 9.2 RFP 구조 분석
- 페이지 단위 텍스트 추출
- 표/섹션/머리글 구조 인식
- 문서별 원문 근거 연결

### 9.3 요구사항 자동 분해
각 요구사항을 다음 구조로 저장한다.
- 요구사항 ID
- 문서/페이지 위치
- 원문
- 분류(기능/비기능/일정/인력/실적/보안/평가/산출물 등)
- 필수 여부
- 증빙 필요 여부
- 해석 요약
- 제안 포인트
- 구현 방향
- 산출물
- 리스크
- 질의 필요 여부
- 신뢰도/검토 상태

### 9.4 누락 탐지
- 필수 제출물 누락
- 실적/인력/증빙 누락
- 평가항목 반영 누락 가능성
- 일정/범위 충돌 가능성
- 모호한 표현 감지

### 9.5 검토/승인
- AI 분석 결과에 대한 사람 검토
- 수정 이력 저장
- 승인/보류/추가검토 상태

### 9.6 유사 제안 재사용
- 과거 제안서 검색
- 유사 요구사항 매칭
- 실적 문구/산출물 구조 재활용

## 10. 신뢰성 설계 원칙
이 프로젝트의 성패는 신뢰성에 달려 있으므로, 다음을 제품 레벨 원칙으로 고정한다.

### 10.1 근거 기반 응답
모든 분석 결과는 가능한 한 다음과 연결한다.
- 원문 인용 위치
- 페이지 번호
- 조항 번호
- 관련 문단

### 10.2 출력 계층 구분
분석 결과는 반드시 다음으로 나눠 표시한다.
- **확정 정보**
- **추론 정보**
- **검토 필요 정보**

### 10.3 구조화 분석 고정 포맷
각 요구사항 분석은 아래 형식을 따른다.
1. 요구사항 원문
2. 발주처 의도
3. 제안서 반영 포인트
4. 구현 방향
5. 산출물
6. 차별화 포인트
7. 리스크
8. 질의 필요 여부

### 10.4 인간 승인 흐름
- AI 1차 분석
- 실무자 검토
- 승인/수정
- 제안서 반영

## 17. 현재 합의된 방향
- 이 제품은 제안서 자동작성기가 아니다.
- 이 제품은 **근거 기반 요구사항 분석 + 누락 방지 + 수주 운영** 시스템이다.
- 가장 중요한 차별점은 **신뢰성**이다.
- 제품의 중심은 **원문 근거, 구조화 분석, 누락 탐지, 사람 검토**다.
- 개발은 단계적으로, 작은 단위로 진행한다.
