# BidOps_UI_GUIDELINES (Updated)

## 1. 문서 목적
이 문서는 BidOps의 화면 기획, 디자인, 퍼블리싱, 프론트엔드 구현이 같은 기준으로 움직이도록 하기 위한 UI 기준서다. 목적은 “예쁜 화면”이 아니라 **업무형 B2B SaaS로서 신뢰감 있고, 정보가 잘 읽히며, 검토와 승인 흐름이 명확한 화면 체계**를 만드는 것이다.

---

## 2. UI 설계 원칙
### 2.1 최우선 원칙
1. 정확성이 시각적 화려함보다 우선한다.
2. 정보 밀도가 높아도 구조가 명확해야 한다.
3. 사용자는 항상 “현재 상태”, “다음 액션”, “근거 위치”를 즉시 이해할 수 있어야 한다.
4. 분석 결과는 사람이 검토하고 수정할 수 있는 형태로 보여야 한다.
5. 중요한 정보는 문장보다 **표, 배지, 상태, 그룹 구조**로 드러나야 한다.
6. AI 분석 상태와 사람 검토 상태를 화면에서 명확히 구분해야 한다.

### 2.2 톤앤매너
- 키워드: 신뢰감, 정돈감, 명료성, 업무성, 차분함
- 지양: 과한 애니메이션, 과한 색상, 소비자 앱 같은 감성형 UX
- 레퍼런스 감각: Notion + 업무용 Admin Dashboard + 문서 검토 도구

### 2.3 디자인 방향
- 좌측 네비게이션 + 상단 컨텍스트 바 + 본문 작업 영역 구조
- 프로젝트/문서/요구사항/체크리스트가 정보 구조의 핵심 축
- 표 기반 화면과 상세 패널 화면이 자연스럽게 연결되어야 함
- PDF 원문 근거와 분석 결과를 빠르게 오갈 수 있어야 함

## 4. 레이아웃 기준
### 4.1 기본 레이아웃
- 좌측 사이드바: 주요 메뉴, 프로젝트 전환
- 상단 헤더: 현재 프로젝트명, 검색, 사용자 메뉴
- 본문: 리스트/상세/보조패널 3영역까지 허용
- 기본 최대 너비: 지나치게 넓지 않게 유지
- 페이지 상단에는 항상 제목, 설명, 주요 액션 버튼 노출

### 4.2 화면 유형
#### A. 목록형 화면
예: 프로젝트 목록, 문서 목록, 요구사항 목록
- 상단 제목/설명
- 필터 바
- 주요 액션 버튼
- 데이터 테이블
- 우측 또는 하단 상세 미리보기 가능

#### B. 상세 분석 화면
예: 요구사항 상세
- 좌측: 원문/근거/PDF 연결
- 중앙: 구조화 분석 내용
- 우측: 검토 상태, 검토 이력, 액션 버튼

#### C. 운영형 대시보드
예: 프로젝트 상세, 분석 결과 대시보드
- 요약 카드
- 진행 상태
- 누락 위험 요약
- 최근 변경사항
- 다음 액션

### 4.3 여백과 밀도
- 섹션 간 여백은 넉넉하게
- 정보량이 많은 화면은 카드 분리보다 **그리드 정렬** 우선
- 컴포넌트 간 간격은 일관되게
- 표 안에서 텍스트가 지나치게 답답하지 않도록 행 높이 확보

## 5. 색상 및 시각 규칙
### 5.1 기본 방향
- 메인 컬러는 신뢰감을 주는 블루/네이비 계열
- 배경은 흰색/매우 연한 회색 중심
- 성공/주의/오류/검토 필요 상태는 명확히 구분

### 5.2 권장 상태 색상
- 기본: 회색/네이비
- 진행중: 파랑
- 승인: 초록
- 검토 필요: 주황
- 위험/누락: 빨강
- 보류: 회색

### 5.3 색상 사용 원칙
- 의미 없는 장식용 색상 사용 금지
- 상태 색상은 텍스트, 배지, 아이콘에 일관되게 적용
- 한 화면에서 과도한 강조 색상 혼용 금지

## 6. 타이포그래피 기준
### 6.1 기본 원칙
- 제목/부제/본문/보조텍스트 위계를 명확히 한다.
- 긴 본문보다 표, 요약 문장, 라벨 중심 구조를 선호한다.
- 중요한 값과 상태는 굵기와 배지로 드러낸다.

### 6.2 권장 계층
- 페이지 타이틀
- 섹션 타이틀
- 카드 타이틀
- 본문
- 보조 설명
- 캡션/라벨

### 6.3 문장 스타일
- 관리자/업무용 서비스에 맞는 짧고 명령형/설명형 문장 사용
- 모호한 마케팅 문구 금지
- 버튼/라벨 텍스트는 최대한 짧고 명확하게

## 7. 핵심 컴포넌트 규칙
### 7.1 버튼
- Primary: 현재 화면의 가장 중요한 실행
- Secondary: 보조 실행
- Tertiary/Text: 낮은 우선순위
- Danger: 삭제/위험 작업

원칙:
- 한 화면에 Primary는 보통 1개
- 버튼 텍스트는 동사형으로 명확하게
- 예: “분석 시작”, “검토 승인”, “근거 보기”, “저장”

### 7.2 배지(Status Badge)
배지는 BidOps에서 매우 중요하다.

표시 예:
- 분석 완료
- 검토 필요
- 승인됨
- 누락 위험
- 질의 필요
- 확정 정보
- 추론 정보

### 7.3 테이블
BidOps의 핵심 UI는 테이블이다.

원칙:
- 첫 열은 식별자/제목
- 짧은 상태 컬럼은 좁게
- 설명 컬럼은 넓게
- 정렬, 필터, 검색 필수
- 행 클릭 시 상세 패널/상세 페이지 이동
- 고정 헤더 권장
- 다중선택은 필요한 화면에만 허용

### 7.4 필터 바
- 키워드 검색
- 분류 필터
- 분석 상태 필터
- 검토 상태 필터
- Fact Level 필터
- 문서 유형 필터
- 정렬
- 초기화 버튼

### 7.5 카드
카드는 요약용으로만 쓴다.
- 총 요구사항 수
- 검토 필요 수
- 누락 위험 수
- 질의 필요 수
- 문서 분석 상태

### 7.6 상세 패널
리스트와 함께 쓰이는 상세 패널은 빠른 검토를 위해 중요하다.

포함 항목:
- 원문
- 분석 요약
- 검토 상태
- 근거 페이지
- 수정/승인 액션

### 7.7 탭
탭은 화면 전환이 아니라 **같은 맥락 안의 하위 정보 그룹화**에만 쓴다.

예:
- 원문
- 구조화 분석
- 검토
- 리스크
- 이력

### 7.8 모달
모달은 단순 확인/짧은 입력만 허용
긴 폼, 긴 분석 내용, 다단계 작업은 모달 금지

## 8. 상태(State) 설계
### 8.1 시스템 상태
- 업로드 대기
- 업로드 완료
- 분석 중
- 분석 완료
- 오류
- 재분석 필요

### 8.2 분석/검토 상태
#### Fact Level 배지
- FACT → 확정 정보
- INFERENCE → 추론 정보
- REVIEW_NEEDED → 검토 필요

#### 검토 상태 배지
- NOT_REVIEWED → 미검토
- IN_REVIEW → 검토 중
- APPROVED → 승인
- HOLD → 보류
- NEEDS_UPDATE → 수정 필요

#### 체크리스트 상태 배지
- TODO → 미착수
- IN_PROGRESS → 진행중
- DONE → 완료
- BLOCKED → 차단됨

#### 위험도 배지
- NONE → 위험 없음
- LOW → 낮음
- MEDIUM → 보통
- HIGH → 높음

### 8.3 화면 상태
- 기본
- 빈 상태(Empty)
- 로딩
- 오류
- 권한 없음
- 결과 없음

### 8.4 Empty State 원칙
- 왜 비어 있는지 설명
- 사용자가 해야 할 다음 액션 안내
- 빈 상태에서도 핵심 CTA 제공

예:
“아직 업로드된 RFP가 없습니다. PDF를 업로드하면 요구사항 분석을 시작할 수 있습니다.”

## 9. 핵심 화면별 가이드
### 9.1 프로젝트 목록
목적:
- 사용자가 자신의 입찰/제안 프로젝트를 빠르게 찾고 상태를 파악

포함:
- 프로젝트명
- 고객사/사업명
- 상태
- 최근 수정일
- 문서 수
- 요구사항 수
- 누락 위험 여부

핵심 액션:
- 새 프로젝트 생성
- 검색
- 필터
- 프로젝트 진입

### 9.2 프로젝트 상세 / 문서 업로드
목적:
- 프로젝트 개요 확인 및 문서 관리

포함:
- 프로젝트 개요 카드
- 업로드된 문서 목록
- 문서 유형 태그
- 분석 상태
- 분석 시작 버튼
- 최근 이력

핵심 액션:
- 문서 업로드
- 분석 시작
- 문서 상세 보기
- 버전 비교

### 9.3 RFP 분석 결과 대시보드
목적:
- 분석 결과를 한눈에 이해하고 다음 액션을 결정

포함:
- 요구사항 총량
- 분류별 개수
- 필수 제출물 요약
- 누락 위험 요약
- 질의 필요 항목
- 최근 검토 항목
- 원문 연결

핵심 액션:
- 요구사항 목록 이동
- 체크리스트 이동
- 검토 시작
- 리스크 항목 보기

### 9.4 요구사항 목록
목적:
- 모든 요구사항을 필터링/정렬/검색하며 검토

권장 컬럼:
- ID
- 분류
- 요구사항 제목/요약
- 필수 여부
- 증빙 여부
- 분석 상태
- 검토 상태
- Fact Level
- 질의 필요 여부
- 근거 페이지

### 9.5 요구사항 상세 분석
목적:
- 한 요구사항을 깊게 검토하고 승인/수정

구성:
- 원문
- 발주처 의도
- 제안 반영 포인트
- 구현 방향
- 산출물
- 차별화 포인트
- 리스크
- 질의 필요 여부
- 근거 위치
- 변경 이력
- 검토 상태
- 검토 코멘트

권장 탭/영역:
- 분석 탭
- 검토 탭
- 근거 탭
- 이력 탭

### 9.6 누락/증빙 체크리스트
목적:
- 제출 이전 리스크를 줄이기 위한 확인 화면

구성:
- 체크 항목
- 근거
- 현재 상태
- 위험도
- 담당자
- 조치 필요 여부
- 마감 상태

## 10. 반응형 기준
BidOps는 데스크톱 우선이다.
- 1차 목표: 노트북/데스크톱 최적화
- 태블릿: 제한적 대응
- 모바일: 조회 중심 최소 대응

모바일에서 작성/검토 업무 전체를 수행하게 설계하지 않는다.

## 12. UX 금지사항
1. 한 화면에 Primary 버튼 여러 개 남발 금지
2. 근거 정보 없는 분석 결과 단독 노출 금지
3. 과한 카드 분할로 정보 흐름 끊기 금지
4. 모달 안에 긴 문서 넣기 금지
5. 상태 색상 의미를 화면마다 바꾸기 금지
6. 리스트와 상세 액션이 분리돼 사용자가 길을 잃는 구조 금지
7. 마케팅성 문구로 업무 시스템의 정확성을 흐리게 만드는 표현 금지
8. AI 분석 결과와 사람 검토 결과를 한 카드 안에 모호하게 섞어 표시하는 구성 금지

## 14. 최종 원칙
BidOps UI의 목표는 “멋진 화면”이 아니라 **신뢰할 수 있는 업무 처리 화면**이다.

따라서 UI는 다음 질문에 항상 답해야 한다.
- 지금 내가 보고 있는 정보는 무엇인가?
- 이 정보는 AI 분석 결과인가, 사람 검토 결과인가?
- 원문 근거는 어디에 있는가?
- 다음 액션은 무엇인가?
